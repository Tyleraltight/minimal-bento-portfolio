# 避坑指南 (Pitfalls & Guidelines)

## 1. 浏览器自动化滥用警告
* **场景**: 尝试抓取网页元素或页面截图。
* **反思**: 先前在获取网页封面图时，不加判断地调用了重型的 `browser_subagent`，试图让其自动截图并寻找DOM。这导致了不必要的等待，且因执行跳过或失败使得过程假死（"do nothing"）。
* **准则**: 
  - **绝对禁止**在非必要场景下调用浏览器自动化工具。
  - **优先使用**轻量级命令（如 `curl`, `Invoke-WebRequest`）抓取源代码，通过正则或文本匹配提取所需内容。
  - **优先依赖**用户已提供的信息和截图，不要擅作主张去重复获取。
  - 仅在需要处理复杂 JS 渲染、动态交互或严格的视觉回归测试时，才在**征得同意**的前提下使用浏览器自动化。

## 2. 计划与执行的配置冲突（Auto-Approval 陷阱）
* **场景**: 收到“先写计划，看了再说”的强指令，但环境配置为 `approval_mode: "full-auto"`。
* **反思**: 以为写了计划并将其输出到 Artifact 就会暂停等待用户确认，但底层配置自动放行了计划，导致我在未得到用户实际回复的情况下直接开始改代码，引发用户的反感。
* **准则**:
  - 当检测到自动放行（System Message: user automatically approved...）但用户的文字要求强硬等待时，必须**在此刻硬性停止执行**（例如通过结束当前回答让用户说话），而不是自顾自地开始执行被系统静默同意的计划。

## 3. 多重替换（Multi-replace）导致括号丢失的致命失误
* **场景**: 替换 JSX 结构中含有条件渲染 `{condition && ( ... )}` 的深层嵌套区块。
* **反思**: 在替换大段 JSX 代码时，为了修改内部标签，不小心吞掉了末尾的条件闭合括号 `)}`，导致了相邻 JSX 元素未被包裹的 Babel 解析错误。
* **准则**:
  - 替换 JSX 大段结构前，必须极度小心审查包裹此区块的**作用域范围**（是否在 `{}` 判断内）。
  - 使用 `multi_replace_file_content` 时，替换前后的开闭合标签（包括大括号、圆括号）必须严格一一对应计算。不要贪图大块替换，应尽量精确定位到需要改动的最小节点。
